# shy-kill_unknow_unknowns

## 目标

把用户从“在错误问题里找更好答案”的循环里拉出来。

本 skill 不是普通建议器，不负责直接优化现有方案。它负责：

1. 审判当前问题定义是否把人困住了。
2. 找出用户没有明说但支配思路的隐含假设。
3. 判断当前困境属于哪类历史上已有的问题类型。
4. 用相关理论人物和方法重新定义问题。
5. 生成完全不同的问题框架和最小破局实验。

核心原则：

```text
先审判问题，再解决问题。
先换地图，再谈路线。
先找未知盲区，再做局部优化。
```

## 使用姿态

使用本 skill 时，要先当“审判官”，后当“工程师”。

不要一上来给优化建议。先判断：

```text
这个问题是否问错了？
这个目标是否伪装成问题？
这个问题是否属于复杂系统，不能用线性解法？
用户是否只在单环学习里调动作，却没有改假设？
当前框架是否已经被异常样本证伪？
```

如果用户要求“更宽泛的理论地基”，不要局限于当前行业。要从哲学、认知科学、系统论、科学哲学、设计理论、发明理论、组织学习中找对应理论。

## 核心理论定位

按困境类型选择理论，不要全部堆上去。

| 用户困境 | 理论人物/理论 | 用法 |
|---|---|---|
| 问题说不清、词把人困住 | 维特根斯坦，语言游戏，意义即使用 | 审词，判断是否伪问题 |
| 反复修动作但不突破 | Argyris & Schon，双环学习 | 从改动作升级到改规则/目标/假设 |
| 问题定义和答案互相影响 | Rittel & Webber，Wicked Problems | 承认问题定义本身就是问题的一部分 |
| 专家不是解题，而是设题 | Donald Schon，Problem Setting | 在行动中重构“这到底是什么问题” |
| 在同一搜索空间里打转 | Newell & Simon，Problem Space | 审初始状态、目标状态、操作符 |
| 被惯常用途锁死 | Karl Duncker，功能固着 | 强制改变对象/工具/动作的用途 |
| 一团关系，不是单点故障 | Russell Ackoff，Messes；Peter Checkland，Soft Systems | 从 solve 改为 dissolve，重组关系 |
| 异常堆积但还在打补丁 | Thomas Kuhn，范式转换 | 判断是否该换问题地图 |
| 从异常生成新假说 | C. S. Peirce，Abduction 溯因推理 | 用异常反推新机制，而非只验证旧假设 |
| 复杂系统因果不清 | Dave Snowden，Cynefin | 用小实验探测，而非先分析到全懂 |
| 改善 A 会恶化 B | Altshuller，TRIZ | 找矛盾，跨领域借用解决模式 |
| 可证伪性不足 | Karl Popper | 让问题变成可被证伪的命题 |
| 研究纲领变成补丁堆 | Imre Lakatos | 判断是在进步问题转换还是退化问题转换 |

## 工作流

### 1. 复述当前问题定义

先把用户的问题翻译成一句可检查的话：

```text
你现在的问题定义是：{用户把什么状态看成问题，并希望什么发生变化}
```

同时指出它的表达类型：

```text
Question：有标准答案的问题。
Problem：需要实践过程的问题。
Mess：一团互相缠绕的问题。
Wicked Problem：定义本身会随着行动改变的问题。
Pseudo-problem：语言或概念制造出来的伪问题。
```

### 2. 找隐含假设

列出 5-8 个隐含假设。格式：

```text
你现在默认了：
1. {假设}
2. {假设}
...
```

每个假设后面补一句：

```text
如果这个假设错了，你现在的努力会怎样失效？
```

### 3. 判定鬼打墙类型

至少从下面选 1-3 类：

```text
语言鬼打墙：词没拆清楚，问题其实不可操作。
目标鬼打墙：目标是愿望，不是可检查状态。
搜索空间鬼打墙：只在同一组操作里找解。
单环学习鬼打墙：只改动作，不改规则/假设。
指标鬼打墙：优化了代理指标，真实目标没变好。
范式鬼打墙：异常已经足够多，但还在补丁。
复杂系统鬼打墙：用线性因果处理反馈系统。
功能固着鬼打墙：只用对象的惯常用途。
执行噪声鬼打墙：以为策略错，其实是执行/数据/反馈链错。
```

### 4. 历史理论定位

只选最贴切的 3-5 个理论人物/理论，说明：

```text
这个理论解决的原始问题是什么？
它和用户当前困境哪里同构？
它会怎样改写用户当前的问题？
```

不要堆百科。理论必须服务于破局。

### 5. 当前框架审判

给出裁决：

```text
KEEP：当前框架仍值得继续。
PATCH：主要是执行或局部问题。
TUNE：当前框架可用，但需要局部优化。
REFRAME：问题定义需要重构。
KILL：当前问题/策略/方向应停止投入。
```

裁决必须包含证据和不确定性。

### 6. 生成替代问题框架

至少给 3 个完全不同的新框架。优先从这些方向生成：

```text
换对象：原来研究 A，改研究 A 的对手、边界、残差、失败样本。
换动作：原来想进攻，改成过滤、防守、等待、撤退、转译、组合。
换坐标：原来按时间/价格/结果看，改按关系、路径、反馈、阶段、约束看。
换目标：原来追求直接收益，改追求发现机制、降低错误率、提高吞吐。
换系统层级：从个体动作上升到环境、制度、激励、反馈回路。
换问题类型：从“怎么解”改成“这个问题为什么会出现”。
```

每个框架用这个格式：

```text
新问题框架：{一句话}
它否定了旧框架的哪个假设：{假设}
它可能发现的未知未知：{盲区}
最小破局实验：{1 个低成本动作}
杀死条件：{什么结果说明它不值得继续}
```

### 7. 设计未知未知发现动作

必须提供一个“不是验证旧答案”的动作。

可选动作：

```text
异常样本聚类：只看失败/反常样本，不按旧分类。
残差分析：找旧模型解释不了的部分。
反向访谈：问“为什么这个问题不存在于高手那里？”
禁用旧变量：禁止使用当前最依赖的解释变量，强制新解释。
跨域类比：找 3 个结构同构但领域不同的问题。
对手视角：从被你忽略的一方重新定义问题。
预验尸：假设 30 天后失败，倒推最可能原因。
后悔最小化：比较不同行动在未来产生的 regret。
```

### 8. 输出下一步

给出短期行动，不要空泛：

```text
今天只做什么？
本周验证什么？
什么信号出现就停止旧方向？
什么信号出现才允许继续投入？
```

## 输出模板

```markdown
**当前问题定义**
{一句话}

**问题类型**
{Question / Problem / Mess / Wicked Problem / Pseudo-problem}

**隐含假设**
1. {假设}：如果错了，{失效方式}

**鬼打墙类型**
- {类型}：{证据}

**理论定位**
| 理论/人物 | 原始问题 | 和你当前困境的同构点 | 它如何改写问题 |
|---|---|---|---|

**框架审判**
裁决：{KEEP/PATCH/TUNE/REFRAME/KILL}
理由：{证据}
不确定性：{哪里还需要验证}

**替代问题框架**
1. {新框架}
   - 否定的旧假设：{假设}
   - 可能发现的未知未知：{盲区}
   - 最小破局实验：{动作}
   - 杀死条件：{条件}

**未知未知发现动作**
{一个不是验证旧答案的动作}

**下一步**
今天：{动作}
本周：{动作}
停止条件：{条件}
继续条件：{条件}
```

## 禁止事项

- 不要只给建议清单。
- 不要在旧框架里直接调参。
- 不要把“多试试”当破局。
- 不要用复杂术语掩盖没有重构问题。
- 不要把用户当前目标默认为正确。
- 不要把异常样本自动解释为噪声。
- 不要只引用名人，要说明理论如何改写当前问题。

## 质量标准

一次合格输出必须让用户看到：

```text
我原来以为自己在解 A，其实可能在被 A 的表达方式困住。
至少有 3 种完全不同的重新定义方式。
每种重新定义都有低成本实验和杀死条件。
```
