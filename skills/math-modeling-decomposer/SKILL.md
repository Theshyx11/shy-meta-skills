# math-modeling-decomposer

## 目标

把现实问题转化为可计算、可验证、可审计的数学模型。

本 skill 不是数学题求解器，也不是公式百科。它要做的是：

```text
现实语言
→ 数学对象
→ 变量表
→ 关系结构
→ 目标函数
→ 约束与不确定性
→ 数学领域定位
→ 理论/公式检索
→ 第一版计算
→ 验证和误用边界
```

核心定义：

```text
数学功底 = 把现实问题转化成可计算结构，并能审计这个转化是否失真的能力。
```

## 使用边界

使用本 skill：

- 用户想把现实问题量化、数学化、模型化。
- 用户问某个业务、交易、策略、学习或决策问题“怎么算”。
- 用户想知道这个问题对应哪个数学分支、理论或公式。
- 用户想诊断自己缺哪类数学能力。
- 用户担心数字、变量、代理指标或模型在误导自己。

不要使用本 skill：

- 用户只问单个数学定理的百科解释。
- 用户只要计算一个已给公式的数值。
- 用户只要代码实现，不需要建模解释。
- 用户要理论源流和历史人物优先时，改用 `shy-theory-grounding`。
- 用户要量化交易变量的深度微观结构解释时，可优先用 `quant-variable-decomposer`，必要时再用本 skill 做通用数学建模。

## 硬规则

1. 每一个变量名、代码字段、公式参数都必须用中文解释。
2. 不要只给抽象框架，必须给出第一版可计算公式或计算路径。
3. 不要把所有东西都数字化；必须区分可观测变量、代理变量和潜变量。
4. 不要迷信数字；必须说明数字可能在哪里骗人。
5. 理论/公式检索必须说明适用条件，不能只贴公式名。
6. 如果需要外部检索而当前没有检索，必须标注“待检索确认”。
7. 输出必须包含“最小验证动作”，让用户能检验模型是否有用。

## 十步工作流

### 1. 问题类型识别

先判断用户问题属于哪类数学问题，可以多选：

```text
预测问题：估计未来会发生什么。
分类问题：判断对象属于哪一类。
估值问题：判断某个对象值多少钱。
优化问题：在约束下最大化或最小化目标。
控制问题：通过行动改变系统状态。
因果问题：判断 X 是否导致 Y。
博弈问题：多个参与者互相影响。
排队/库存问题：资源、等待、容量和服务率。
风险问题：控制亏损、波动、尾部事件或失败概率。
```

### 2. 数学对象定义

明确模型里的对象是什么：

```text
对象：被建模的实体，例如用户、订单、策略、资产、内容、市场、项目。
状态：对象在某个时间点的可描述状态。
事件：状态发生变化的触发点。
边界：决定结果切换的阈值或约束。
时间尺度：秒、天、周、生命周期或项目周期。
```

### 3. 变量表

必须把变量分成六类：

```text
目标变量：最终想解释、预测或优化的量。
可观测变量：能直接从数据、记录、盘口、行为或报表中测到的量。
代理变量：不能直接测真实目标时，用来间接代表目标的量。
潜变量：真实存在但暂时测不到，只能通过模型推断的量。
控制变量：用户能主动改变的量。
约束变量：不能违反的限制条件。
```

变量解释模板：

```text
变量名：
中文含义：
公式或计算方式：
每一项解释：
单位：
正负方向或大小含义：
为什么需要它：
它不代表什么：
常见误用：
```

### 4. 关系结构

判断变量之间是什么关系：

```text
线性关系：一个变量按固定比例影响另一个变量。
非线性关系：边际影响会变化。
阈值关系：超过某个边界后机制改变。
动态关系：当前状态依赖过去路径。
反馈关系：结果反过来影响原因。
因果关系：干预 X 会改变 Y。
约束关系：变量必须满足某些限制。
随机关系：结果只以概率形式出现。
```

### 5. 目标函数

把“好坏”写成数学目标：

```text
最大化：收益、胜率、利润、增长、效率、效用。
最小化：成本、风险、时间、波动、错误率、机会损失。
权衡：收益 - 风险 - 成本 - 机会成本。
```

目标函数模板：

```text
objective = benefit - cost - risk_penalty - opportunity_cost
```

`objective`：目标函数，表示当前决策整体值得做的程度。  
`benefit`：收益项。  
`cost`：直接成本。  
`risk_penalty`：风险惩罚。  
`opportunity_cost`：机会成本。

### 6. 不确定性结构

拆出不确定性来源：

```text
噪声：随机波动。
样本不足：观察数量太少。
测量误差：数据本身不准。
代理变量误差：代理变量没有真正代表目标。
分布漂移：未来分布和历史不同。
对手行为：其他参与者会反应。
反馈延迟：行动效果延后出现。
模型错设：公式结构本身不对。
```

### 7. 数学领域定位

根据问题结构定位数学分支：

```text
概率论：不确定事件、条件概率、贝叶斯更新。
统计推断：样本、估计、置信区间、假设检验。
优化理论：目标函数、约束、最优解。
线性代数：多变量关系、因子、矩阵、降维。
微积分：边际变化、速度、敏感性。
随机过程：时间序列、路径、首次穿越、扩散。
动态系统：状态转移、惯性、反馈、稳定性。
博弈论：对手、策略互动、均衡。
信息论：信息增益、熵、信号和噪声。
极值理论：尾部风险、极端低点或高点。
排队论：容量、等待、服务率、拥堵。
市场微观结构：订单簿、流动性、冲击、恢复。
```

### 8. 理论/公式检索层

把数学领域转成可检索的理论和公式。能联网时应检索；不能联网时输出检索词并标注“待检索确认”。

常见检索入口示例：

```text

回撤分布：drawdown distribution
市场冲击：market impact model
订单簿恢复：order book resilience
排队模型：queueing theory
```

### 10. 验证和误用边界

必须回答：

```text

反例是什么？
哪些变量可能只是代理变量？
模型优化后会不会失效？

```

## 数学功底缺口诊断

当用户问“我缺什么数学功底”或明显卡在某一步时，用下面映射诊断：

```text
不知道变量怎么定义：缺建模抽象能力。
不知道时间怎么进入模型：缺动态系统或随机过程能力。
不知道样本少怎么办：缺统计推断能力。
不知道为什么回测好实盘差：缺过拟合、分布漂移和样本外验证能力。
不知道如何做决策：缺优化和期望效用能力。
不知道多变量怎么处理：缺线性代数和因子建模能力。
不知道极端情况怎么算：缺尾部风险和极值理论能力。
不知道对手会怎么反应：缺博弈论或市场微观结构能力。
不知道数字为什么误导：缺测量误差、代理变量和因果推断能力。
```
